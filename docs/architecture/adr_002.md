# ADR 002: State management – Zustand + Context

| Status | accepted |
|--------|----------|
| Date | 2026-05-01 |
| Autor | IT Architect |

## Kontekst

Aplikacja Introsphere wymaga zarządzania stanem na kilku poziomach:
- Globalny stan autoryzacji (użytkownik, sesja)
- Stan sesji refleksji (wybrana emocja, journey, dzien, odpowiedzi)
- Ustawienia motywu i preferencji interfejsu
- Stan widoku (płytkie dane UI: sidebar, modal, loading)

Rozważane opcje:
1. Tylko React Context + useState/useReducer – wbudowane, brak dodatkowych zależności
2. Redux Toolkit – pełne narzędzie, duży boilerplate, RTK Query do API
3. Zustand – minimalistyczny, lekki, bez boilerplate
4. Jotai / Valtio – atomowe podejście, ale mniejszy ekosystem
5. MobX – opiniowany, ale starszy ekosystem

Kryteria doboru:
- Minimalny bundle size (aplikacja PWA dla Gen Z – performance)
- Łatwość nauki i utrzymania (zespół ma 1 seniora + juniorów)
- Dobry TypeScript support (strict mode)
- Proste persystowanie (localStorage/sync)
- Skalowalność do średniej wielkości aplikacji (ok. 30–50 komponentów)

## Decyzja

Wybór **Zustand** jako głównego store dla stanu aplikacji w połączeniu z **React Context** jako dostawcą providerów i warstwą integracyjną.

Wzorzec:
- AuthStore, SessionStore, ThemeStore jako osobne storey Zustand
- React Context używane do:
  - Dostarczania store do drzewa komponentów (provider pattern)
  - Izolacji części stanu wymagającego subskrypcji selektywnej
  - Globalnych providerów takich jak AuthProvider, ThemeProvider

## Konsekwencje

### Pozytywne
- Bardzo mały rozmiar bundle (~1-2 kB gzipped)
- Proste API (get/set) – krótka krzywa uczenia
- Built-in persystencja (middleware) – tanie zapisywanie preferencji i sesji w localStorage
- Łatwe testowanie (stores jako zwykłe obiekty)
- Dobre TypeScript wsparcie (bez dodatkowych konfiguracji)
- Brak boilerplate w komponentach (nie trzeba opakowywać w connect/HOC)

### Negatywne
- Brak wbudowanych narzędzi do devtools na poziomie Redux DevTools (choć Zustand ma podstawowy devtools)
- Brak gotowego mechanizmu do podziału store na "slice'ów" z automatycznym łączeniem (jak RTK)
- Mniejsza społeczność/wsparcie niż Redux (choć Zustand jest popularny)
- Trudniej o zaawansowane selektywne subskrypcje (wymaga ręcznego podziału store lub selectorów)

## Szczegóły implementacji

### Przykładowy store (TypeScript strict)

```typescript
interface SessionStore {
  currentMood: Emotion | null;
  currentJourneyId: string | null;
  currentDay: number;
  currentQuestion: Question | null;
  responseText: string;
  
  selectMood: (emotion: Emotion) => void;
  startJourney: (journeyId: string) => void;
  nextDay: () => void;
  reset: () => void;
}

export const useSessionStore = create<SessionStore>((set) => ({
  currentMood: null,
  currentJourneyId: null,
  currentDay: 1,
  currentQuestion: null,
  responseText: '',
  
  selectMood: (emotion) => set({ currentMood: emotion }),
  startJourney: (journeyId) => set({ currentJourneyId: journeyId, currentDay: 1 }),
  nextDay: () => set((state) => ({ currentDay: state.currentDay + 1 })),
  reset: () => set({ currentMood: null, currentJourneyId: null, currentDay: 1, currentQuestion: null, responseText: '' }),
}));
```

### Persystencja (middleware)

```typescript
persist.create<SessionStore>(
  (set, get) => ({
    // ...initial state
  }),
  {
    name: 'session-storage',
    partialize: (state) => ({ currentMood: state.currentMood }),
  }
)
```

## Alternatywy

- **Redux Toolkit**: preferowany, gdy aplikacja skomplikowana (wielu aktorów, złożone transakcje, offline queue), ale wyższy koszt wprowadzenia na MVP
- **TanStack Query (React Query)**: jako uzupełnienie do cache'owania po stronie klienta dla danych z API zamiast trzymania ich w globalnym secie
- **Context + useReducer**: dla bardzo małych aplikacji (brak zależności), ale wydajnościowo gorsze przy częstych aktualizacjach
- **Jotai**: alternatywa atomowa, podobna prostota, ale mniejszy ekosystem toolingu

## Migracja

Jeśli w przyszłości potrzeba zmiany:
- Stores są izolowane w `stores/` – łatwa migracja do innego rozwiązania
- Interfejsy w `types/` pozwalają na zachowanie kontraktu
- Testy integracyjne nad API store pozwolą na bezpieczną migrację
- Event sourcing pattern może zostać dodany jako middleware w razie potrzeby audytu zmian
