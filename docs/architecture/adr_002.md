# ADR 002: State management – Zustand + Context

| Status | accepted |
|--------|----------|
| Date | 2026-05-01 |
| Autor | IT Architect |

## Kontekst

Aplikacja Introsphere wymaga zarządzania stanem na kilku poziomach:
- Globalny stan autoryzacji (użytkownik, sesja)
- Stan sesji refleksji (wybrana emocja, journey, dzien, odpowiedzi)
- Ustawienia motywu i preferencji interfejsu
- Stan widoku (płytkie dane UI: sidebar, modal, loading)

Rozważane opcje:
1. Tylko React Context + useState/useReducer – wbudowane, brak dodatkowych zależności
2. Redux Toolkit – pełne narzędzie, duży boilerplate, RTK Query do API
3. Zustand – minimalistyczny, lekki, bez boilerplate
4. Jotai / Valtio – atomowe podejście, ale mniejszy ekosystem
5. MobX – opiniowany, ale starszy ekosystem

Kryteria doboru:
- Minimalny bundle size (aplikacja PWA dla Gen Z – performance)
- Łatwość nauki i utrzymania (zespół ma 1 seniora + juniorów)
- Dobry TypeScript support (strict mode)
- Proste persystowanie (localStorage/sync)
- Skalowalność do średniej wielkości aplikacji (ok. 30–50 komponentów)

## Decyzja

Wybór **Zustand** jako głównego store dla stanu aplikacji w połączeniu z **React Context** jako dostawcą providerów i warstwą integracyjną.

Wzorzec:
- AuthStore, SessionStore, ThemeStore jako osobne storey Zustand
- React Context używane do:
  - Dostarczania store do drzewa komponentów (provider pattern)
  - Izolacji części stanu wymagającego subskrypcji selektywnej
  - Globalnych providerów takich jak AuthProvider, ThemeProvider

## Konsekwencje

### Pozytywne
- Bardzo mały rozmiar bundle (~1-2 kB gzipped)
- Proste API (get/set) – krótka krzywa uczenia
- Built-in persystencja (middleware) – tanie zapisywanie preferencji i sesji w localStorage
- Łatwe testowanie (stores jako zwykłe obiekty)
- Dobre TypeScript wsparcie (bez dodatkowych konfiguracji)
- Brak boilerplate w komponentach (nie trzeba opakowywać w connect/HOC)

### Negatywne
- Brak wbudowanych narzędzi do devtools na poziomie Redux DevTools (choć Zustand ma podstawowy devtools)
- Brak gotowego mechanizmu do podziału store na "slice'ów" z automatycznym łączeniem (jak RTK)
- Mniejsza społeczność/wsparcie niż Redux (choć Zustand jest popularny)
- Trudniej o zaawansowane selektywne subskrypcje (wymaga ręcznego podziału store lub selectorów)

## Szczegóły implementacji (RZECZIWISTE)

### Stores zaktualizowane do stanu faktycznego (2026-05-11)

```typescript
// 1. authStore.ts – zarządza stanem autentykacji (user, session, loading)
interface AuthState {
  user: User | null;
  session: Session | null;
  loading: boolean;
  setUser: (user: User) => void;
  setSession: (session: Session) => void;
  setLoading: (loading: boolean) => void;
}
export const useAuthStore = create<AuthState>()(
  persist((set) => ({ ... }), { name: 'auth-storage' })
);
// ✅ Zaimplementowane – używane w useAuth + ProtectedRoute

// 2. themeStore.ts – zarządza atmosphere + dark mode
interface ThemeStore {
  atmosphere: AtmosphereType;    // 8 dostępnych motywów
  isDark: boolean;               // false = light, true = dark
  setAtmosphere: (a: AtmosphereType) => void;
  setIsDark: (d: boolean) => void;
}
export const useThemeStore = create<ThemeStore>()(
  persist((set) => ({ ... }), { name: 'theme-storage' })
);
// ✅ Zaimplementowane – aplikacja klas .atmosphere-{name} + .dark na <html>

// 3. journeyStore.ts – zarządza lokalnym stanem completed days (localStorage)
interface JourneyStore {
  completedDays: Record<string, number[]>; // { [journeyId]: [1,2,3...] }
  setCompletedDays: (journeyId: string, dayNumber: number) => void;
  getCompletedDays: (journeyId: string) => number[];
  resetJourney: (journeyId: string) => void;
  getCurrentDay: (journeyId: string) => number;
  syncFromDatabase: () => Promise<void>; // synced z calendar_entries (nie user_journey_progress!)
}
export const useJourneyStore = create<JourneyStore>()(
  persist((set, get) => ({ ... }), { name: 'journey-completed-days' })
);
// ✅ Zaimplementowane – ale journey progress liczony z DB (calendar_entries), nie z tej tablicy (array to UI cache)

// 4. sessionStore.ts – LIGHTWEIGHT, NIE pełny session management!
interface SessionState {
  entryDeletedCount: number;          // tylko do trigger refetch po deletion
  incrementEntryDeletedCount: () => void;
}
export const useSessionStore = create<SessionState>((set) => ({
  entryDeletedCount: 0,
  incrementEntryDeletedCount: () => set((s) => ({ entryDeletedCount: s.entryDeletedCount + 1 })),
}));
// ⚠️ NIE zaimplementowano planowanego sessionStore z:
//   currentMood, currentJourneyId, currentDay, currentQuestion, responseText, selectMood(), startJourney(), nextDay(), reset()
// Rzeczywista logika sesji jest rozproszona w page components (Session.tsx, EmotionReflection.tsx) + localStorage (journeyStore)
```

**UWAGA:** Oryginalny plan ADR-002 zakładał `SessionStore` jako centralne miejsce dla stanu sesji. W rzeczywistości:

- **Session selection** (emotion pick) → stan lokalny w `Session.tsx` (`selectedEmotion` state)
- **Reflection session** (response text) → stan lokalny w `EmotionReflection.tsx` (`responseText` state)
- **Journey progress** → `journeyStore.ts` (completedDays w localStorage + sync z DB)
- **Deletion notifications** → `sessionStore.ts` (tylko counter)

To oznacza brak centralnego state managera dla sesji – co jest **akceptowalne dla MVP**, ale ogranicza możliwość复杂的 session flow (np. pause/resume, multi-step journeys w jednej sesji).

### Persystencja (middleware)

```typescript
// persystencja w 3 z 4 store'ów:
persist((set, get) => ({ ... }), { name: 'auth-storage' })  // authStore
persist((set, get) => ({ ... }), { name: 'theme-storage' }) // themeStore
persist((set, get) => ({ ... }), { name: 'journey-completed-days' }) // journeyStore

// sessionStore – brak persystencji (niepotrzebne – licznik tymczasowy)
```

### Konsekwencje uproszczenia

**Pozytywne:**
- Mniej kodu – brak boilerplate store'ów które i tak nie byłyby używane w pełni
- Lokalny stan w page components jest wystarczający dla single-question flow
- Mniej zależności między komponentami (brak shared session state)

**Negatywne:**
- Brak możliwości paused sessions (np. opuściłeś /emotion-reflection, wróć później – tracisz response)
- Brak globalnego session timeout / cleanup
- Brak możliwości "resume gdzie zostawiłeś" między journa a session
- Testing harder – stan rozproszony, nie ma jednego store'a do mockowania

### Alternatywy (faktycznie rozważane)

**TanStack Query (React Query)** – wybrano Zustand, ale:
- React Query mógłby zarządzać server state (calendar entries, journeys) z cache'owaniem
- Obecnie: data fetching w page components (useEffect) + ręczny refetch po mutations
- To działa, ale brakuje clever cache invalidation (ręczne invalidate po submit)

**Context + useReducer** – odrzucone:
- Wymagałoby więcej boilerplate
- Provider nesting
- Performance: częste re-rendery całego tree

**Jotai** – podobny do Zustand, ale mniejsze ecosystem – odrzucone.

### Rekomendacje Future

Jeśli aplikacja rośnie:
1. **Dodaj `useSession()` hook** – wrapper wok localStorage session (partial persistence)
2. **Rozważ React Query** – dla wszystkich DB query (calendar, history, stats) – automagiczne caching + invalidation
3. **Stwórz `sessionStore` full** – jeśli dodasz multi-step journeys (dzień 1→7 w jednej sesji)
4. **Wygeneruj Supabase types** – zastąp ręczne `types/database.ts`

## Alternatywy

- **Redux Toolkit**: preferowany, gdy aplikacja skomplikowana (wielu aktorów, złożone transakcje, offline queue), ale wyższy koszt wprowadzenia na MVP
- **TanStack Query (React Query)**: jako uzupełnienie do cache'owania po stronie klienta dla danych z API zamiast trzymania ich w globalnym secie
- **Context + useReducer**: dla bardzo małych aplikacji (brak zależności), ale wydajnościowo gorsze przy częstych aktualizacjach
- **Jotai**: alternatywa atomowa, podobna prostota, ale mniejszy ekosystem toolingu

## Migracja

Jeśli w przyszłości potrzeba zmiany:
- Stores są izolowane w `stores/` – łatwa migracja do innego rozwiązania
- Interfejsy w `types/` pozwalają na zachowanie kontraktu
- Testy integracyjne nad API store pozwolą na bezpieczną migrację
- Event sourcing pattern może zostać dodany jako middleware w razie potrzeby audytu zmian
