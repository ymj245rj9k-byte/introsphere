# ADR 001: Wybór architektury – JAMstack + Supabase BaaS

| Status | accepted |
|--------|----------|
| Date | 2026-05-01 |
| Autor | IT Architect |

## Kontekst

Przy projektowaniu MVP Introsphere (aplikacja journaling dla Gen Z) rozważano różne podejścia do architektury:

1. Tradycyjny backend (Node/Express + PostgreSQL) z pełnym zarządzaniem serwerem
2. Serverless Functions (Vercel/Netlify) z własną warstwą API
3. **JAMstack + BaaS (Supabase)** – statyczny frontend z usługami backendowymi przez Supabase

Wymagania kluczowe:
- Szybki time-to-market (MVP w 4–6 tygodni)
- Żadnego utrzymywania infrastruktury (żadnych serwerów, certyfikatów, skalowania)
- Darmowy / tani tier do startu i testów
- Autentykacja, baza danych i storage gotowe z natęki

## Decyzja

Wybór architektury **JAMstack + Supabase (Backend-as-a-Service)**.

- Frontend: React + Vite (statyczny), hostowany na Vercel
- Backend: Supabase (PostgreSQL, Auth, Storage, Realtime, Edge Functions)
- Stylistyka i UI: Tailwind + Shadcn UI
- State: Zustand + React Context

## Konsekwencje (RZECZIWISTE)

### Pozytywne (zrealizowane)
- ✅ Szybki prototyp – MVP zbudowany w ~15h (plan) + dodatkowe bugfixy
- ✅ Globalny CDN (Vercel planned) – nie wdrożone, ale gotowe do deploy
- ✅ Supabase darmowy tier – używane do 100%
- ✅ Brak konieczności własnych endpointów – wszystkie operacje przez Supabase JS client
- ✅ RLS w Postgres – włączone, ale polityki mogą wymagać dopracowania (uprawnienia)

### Negatywne / Ograniczenia (faktyczne)
- ⚠️ Vendor lock-in na Supabase – migracja wymagałaby dużo pracy
- ⚠️ Mniej kontroli nad custom logic – operacje beam w lib/database.ts, ale brak endpointów API (wszystko client-side)
- ⚠️ Niektóre operacje wymagają własnych funkcji – np. complex statystyki mogłyby potrzebować DB function lub Edge Function
- ⚠️ Schema中有未使用的表 (user_journey_progress, user_responses, user_mood_checkins) – dead weight

### Mitigacje (zastosowane)
- Warstwa danych w `lib/database.ts` – izoluje Supabase SDK (łatwiejsza migracja)
- Type safety częściowo (types/database.ts ręczne, nie generated)
- Separacja: journeys i emocje jako static data (łatwe wyjęcie)
- BrakSupabase Functions/Edge Functions użytych – aplikacja jest czystym clientem


## Alternatywy

- **Serverless Functions + własna baza**: większa kontrola, ale dłuższy time-to-market i utrzymywanie infrastruktury
- **Firebase**: podobny BaaS, ale Postgres (Supabase) lepiej pasuje do relacyjnych danych w tej aplikacji (profilowanie, journeys, kalendarze)
- **Self-hosted Node + Postgres**: największa kontrola, ale wysokie koszty utrzymania i operacji DevOps na etapie MVP
