# ADR 001: Wybór architektury – JAMstack + Supabase BaaS

| Status | accepted |
|--------|----------|
| Date | 2026-05-01 |
| Autor | IT Architect |

## Kontekst

Przy projektowaniu MVP Introsphere (aplikacja journaling dla Gen Z) rozważano różne podejścia do architektury:

1. Tradycyjny backend (Node/Express + PostgreSQL) z pełnym zarządzaniem serwerem
2. Serverless Functions (Vercel/Netlify) z własną warstwą API
3. **JAMstack + BaaS (Supabase)** – statyczny frontend z usługami backendowymi przez Supabase

Wymagania kluczowe:
- Szybki time-to-market (MVP w 4–6 tygodni)
- Żadnego utrzymywania infrastruktury (żadnych serwerów, certyfikatów, skalowania)
- Darmowy / tani tier do startu i testów
- Autentykacja, baza danych i storage gotowe z natęki

## Decyzja

Wybór architektury **JAMstack + Supabase (Backend-as-a-Service)**.

- Frontend: React + Vite (statyczny), hostowany na Vercel
- Backend: Supabase (PostgreSQL, Auth, Storage, Realtime, Edge Functions)
- Stylistyka i UI: Tailwind + Shadcn UI
- State: Zustand + React Context

## Konsekwencje

### Pozytywne
- Szybki prototyp i wdrożenie (brak konfiguracji serwerów)
- Globalny CDN (Vercel) z domyślnym cachowaniem i optymalizacją
- Supabase darmowy tier wystarczający na wczesny etap i spory rozmiar danych
- Brak konieczności implementacji własnych endpointów API dla większości operacji CRUD
- Wbudowane RLS (Row Level Security) w Postgres – bezpieczeństwo na poziomie bazy
- Łatwe przejście na płatne plany Supabase i Vercel w miarę wzrostu

### Negatywne
- Mniejsza kontrola nad niestandardową logiką backendową (wymaga Supabase Functions lub własnego API w przyszłości)
- Vendor lock-in na Supabase (migracja bazy i logiki auth do własnego stacku wymaga pracy)
- Operacje wymagające cięższej obróbki po stronie serwera mogą wymagać osobnych funkcji (Edge/Vercel Functions)
- Limit requestów w darmowym tierze Supabase w ekstremalnych przypadkach (ale odpowiednie dla MVP)

## Mitigacje

- Utrzymywanie warstwy danych przez repository pattern – łatwiejsza migracja w przyszłości, jeśli zajdzie potrzeba
- Type-safe client Supabase – izolacja SDK od reszty kodu (łatwiejsza zmiana w razie potrzeby)
- Separacja auth: jeśli w przyszłości konieczne, można dodać własny provider obok Supabase Auth
- Monitorowanie limitów Supabase od samego początku (dashboard, alerty)

## Alternatywy

- **Serverless Functions + własna baza**: większa kontrola, ale dłuższy time-to-market i utrzymywanie infrastruktury
- **Firebase**: podobny BaaS, ale Postgres (Supabase) lepiej pasuje do relacyjnych danych w tej aplikacji (profilowanie, journeys, kalendarze)
- **Self-hosted Node + Postgres**: największa kontrola, ale wysokie koszty utrzymania i operacji DevOps na etapie MVP
